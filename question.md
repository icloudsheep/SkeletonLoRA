## 已确认事项

1. 后续默认只修改根目录的 SkeletonLoRA 项目。
   `__SHE-LoRA-main/` 与 `_scripts/` 暂作为参考实现，除非明确指定，不作为修改目标。
2. 当前暂不恢复可变 skeleton rank 扫描；先保持代码现状，并补全文档描述。
3. 关注范围限定为：
   - 聚合与解密精确度；
   - 性能，尤其是全加密与部分加密的加密/解密速度；
   - 通信量，尤其是全加密与部分加密的对比；
   - 真实 LoRA 权重下的实验。

## 第三轮代码与数学核对

### 当前对 partial 的可行性分析

设单个 AB 对为：

$$
B \in \mathbb{R}^{d\times r},\qquad A \in \mathbb{R}^{r\times d},\qquad \Delta W=BA
$$

如果同时加密 B 的前 `p%` 行和 A 的前 `p%` 列，则可以把 B、A 分成：

```text
B = [加密行; 明文行]
A = [加密列 | 明文列]
```

服务端可以得到混合结果：

- 加密 B 行 × 明文 A 列：密文结果；
- 加密 B 行 × 加密 A 列：密文结果；
- 明文 B 行 × 加密 A 列：密文结果；
- 明文 B 行 × 明文 A 列：明文结果。

因此，partial 可以在数学上实现，但它不是当前 `half` 分支的简单改名；服务端必须输出明文块和密文块，客户端再按坐标拼成完整结果。只加密 A 的部分列在 CKKS 上通常也可以运行，因为 B 可以作为明文参与密文×明文计算，但其隐私语义和实验对照对象需要单独确认。

### 当前对 skeleton 的可行性分析

skeleton 需要最终聚合矩阵的完整 `r_s` 个列和 `r_s` 个行。partial 下，被选中的行列可能各自包含明文片段和密文片段：

1. 服务端按坐标返回明文片段和密文片段；
2. 客户端只解密密文片段；
3. 客户端按行列坐标拼接出完整 `C_r`、`R_r`；
4. 客户端再执行 CUR 重建。

从 CUR 的线性代数要求看，关键是 `C_r`、`R_r` 和交叉块 `M_r` 的数值完整且坐标一致，并不要求一整行或一整列必须全部是同一种表示。但如果未加密片段直接暴露给服务端，则这是 partial 允许的明文泄露，而不是 CKKS 隐私边界内的全密文保护。

## 第四轮代码与数据结构核对

### 矩形 LoRA 投影问题

当前根项目所有核心路径都以一个 `dim` 表示矩阵维度，并隐含：

```text
B.shape = (dim, rank)
A.shape = (rank, dim)
BA.shape = (dim, dim)
```

但真实模型中的 LoRA 投影不一定是方阵：

```text
B.shape = (out_features, rank)
A.shape = (rank, in_features)
BA.shape = (out_features, in_features)
```

例如 MLP 的 `up_proj/down_proj/gate_proj` 很可能满足 `out_features != in_features`。矩形矩阵仍可使用 CUR：选 `r_row` 个行和 `r_col` 个列，交叉块必须是方阵，通常令 `r_row = r_col = r`，但完整矩阵形状为 `out_features × in_features`。

因此，支持所有投影时不能简单把所有矩阵强行看成同一个 `d×d`，需要明确是：

- 原生支持矩形 B/A/BA 和矩形 CUR；
- 或将矩形矩阵零填充到全局最大方阵后计算；
- 或暂时只选择方阵投影。

### partial 与 skeleton 的表示问题

在 partial 中，服务端可以在不解密的前提下处理带坐标元数据的混合上传包：明文片段直接参与明文运算，密文片段通过公开 context 反序列化后参与同态运算。服务端不需要知道客户端的私有明文内容即可完成逐客户端结果重建和跨客户端求和。

但 skeleton 下发的每一条行/列可能同时包含明文片段和密文片段。客户端需要按坐标合并两种表示，形成完整的骨架行列后再执行 CUR；不能把“明文行”和“密文行”简单视为两条独立向量。

### 当前代码核对

- 当前 `SKELETON_R=16`，并通过 uniform 索引选择 skeleton；可变 rank 扫描仍未恢复。
- 当前 `cur_reconstruct()` 只接受方阵语义下的 `dim`，未显式区分输出维度和输入维度。
- 当前 `load_real_factors()` 只加载方形层的一个 `q_proj`，尚未验证矩形 LoRA key 的方向和形状。
- 当前 skeleton 输出只支持密文分组，没有明文片段与密文片段的坐标级合并协议。
- 当前没有 TensorBoard writer、run 目录、实验恢复或实时进度指标。

## 第五轮数学正确性与实验口径核对

### 关于“partial 加密数量低于 skeleton rank”的说明

按当前比例网格，真实维度为 3200 时：

```text
0.125% → 4 个 B 行、4 个 A 列
0.25%  → 8 个 B 行、8 个 A 列
0.5%   → 16 个 B 行、16 个 A 列
```

因此确实会出现 partial 加密的行数或列数少于 `SKELETON_R=16`。这在数学上不要求报错，因为：

- partial 的数量决定哪些输入因子区域加密；
- skeleton rank 决定最终聚合矩阵采样多少行列；
- 两者是不同维度的控制参数。

当 skeleton 选中的行列跨越 partial 的明文/密文边界时，只需要按坐标拼接完整骨架行列。其影响主要是：低比例下部分 skeleton 内容为明文，实际需要解密的密文元素可能显著少于理论的 `r_s × (out_features + in_features)`。

因此 CSV 不能只记录 skeleton 总元素数，还应记录其中的明文元素数和密文元素数。

### 关于服务端不可解密聚合的说明

服务端可以在不知道密文具体数值的情况下完成聚合。服务端需要知道公开的结构元数据，例如：

```text
客户端 ID、AB 标识、矩阵形状、行列坐标、明文/密文类型、打包分组
```

服务端使用公开 CKKS context 加载密文，执行加法、密文×明文或密文×密文运算，但不持有 secret key。partial 中未加密区域本身会以明文暴露；加密区域的数值仍不可见。

### 关于 CUR 数值稳定性的说明

当前实现要求交叉块 `M` 的矩阵秩严格等于 skeleton rank，并使用：

```text
inv(M)
```

真实权重中即使理论聚合秩不超过 16，也可能出现：

- 实际秩小于 16；
- uniform 索引选出的交叉块降秩；
- CKKS 近似误差导致数值秩判断不稳定；
- `M` 可逆但条件数很大，普通逆放大误差。

因此必须明确是否允许使用 Moore-Penrose 伪逆 `pinv(M)`，并记录奇异值、数值秩和条件数。

## Q114：为什么执行 `python modern_main.py --real` 后终端没有输出？

### 回答

原入口只在全部真实权重读取、CKKS context 创建和实验任务完成后打印最终路径，因此运行期间终端看起来像“没有输出”，实际可能仍在工作。当前真实目录中发现 50 个 safetensors 文件，但配置 `N_CLIENTS=4`，程序只读取 `client_0` 至 `client_3`；这不是自动使用全部 50 个客户端。

预检读取到 4 个配置客户端、104 个合法 AB 标识。默认参数会对每个 AB 执行 38 个任务（明文参考 2 个，外积 18 个，内积 18 个），完整运行约展开 3952 个任务，真实 CKKS 计算可能长时间没有终端反馈。

已在 `modern_main.py` 增加启动、真实文件扫描、AB 发现、维度/槽位检查、context 创建和逐任务的即时进度输出，并使用 `flush=True`；未启动完整真实实验。

## Q115：为什么一轮运行中内存占用暴涨至 16GB+？当前任务是串行还是并行？

### 回答

当前 `modern_main.py` 的实验任务是串行执行：按方法、AB 标识、实验模式、部分加密比例、skeleton 状态逐层循环，没有线程池、进程池或显式并行调度。TenSEAL/底层数值库可能在单个同态运算内部使用原生线程，但这不改变任务级串行性质。

内存暴涨主要来自单个 CKKS 任务的中间对象，而不是同时运行多个实验：

1. `modern_main.py` 会预先把 4 个配置客户端的全部 AB 张量加载到 `pairs_by_identifier` 中；当前共发现 104 个合法 AB。
2. `fe_outer_hybrid.py` 的 `encrypt_upload()` 会为一个客户端、一个 AB、一个模式预先构造所有结果块的 A/B 密文，并把序列化密文全部保存在 `uploads` 中。
3. `aggregate()` 聚合时还会把这些序列化密文反序列化为 CKKS 原生对象，创建乘法、加法和缩放的中间密文，同时保存聚合后的序列化结果；因此上传密文、反序列化对象、运算中间结果和聚合结果会在同一任务阶段重叠。
4. 大型矩形 LoRA 层在 full 或较高部分加密比例下会产生大量块；若矩阵接近 `out_features × in_features` 的大尺寸，块数量和密文总量会按矩阵元素规模增长。
5. TenSEAL/C++ 分配器释放对象后不一定立即把内存归还操作系统，所以 macOS 活动监视器看到的 RSS 可能在任务结束后仍维持高位；这不表示任务并行。

内积路径也可能产生高峰，尤其是加密 B 的逐行/逐元素密文路径：`fe_inner_hybrid.py` 会为结果列或单元素保存大量密文 payload。`rows` 只保存指标，不会保存每个结果矩阵，因此不是主要累积源；主要风险在单个任务的 `uploads`、`aggregate_result` 和 TenSEAL 原生中间对象。

尚未修改内存管理策略，也未再次启动真实实验。

## Q118：当前实现还有哪些值得做的优化？

### 回答

当前已把 TensorBoard 和 CSV 的展示结构收敛到可读状态，结果指标也已经按 `AB index` 连续展示。再往下看，真正还有价值的优化主要不是展示层，而是真实权重的内存路径：

- `fe_data.discover_client_pairs()` 目前会把每个客户端的所有 AB tensor 先读进内存；
- 这对 104 个 AB 的结果展示没问题，但对更大的真实权重 run 仍然会抬高峰值内存；
- 下一步更值得做的是改成“按 AB 发现、按 AB 读取、按 AB 处理”，尽量避免把所有 AB 长驻在 `pairs_by_identifier` 或同类结构中。

另外两项低优先级优化是：

- 降低 `logger.flush()` 频率，减少频繁刷盘；
- 对 `ab_profile` 只保留少量基础特征，避免为展示再额外增加太多图表。
